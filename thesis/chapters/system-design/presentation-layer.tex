\section{The presentation layer}\label{sec: presentation-layer} 
The presentation layer is responsible for rendering the split-view and dispatching user interactions. User interactions include for example highlighting cells on hover, modifying constant values, and editing edges in the flow graph. The system uses a split view, where the worksheet occupies the left side and the flow graph occupies the right side, allowing both to be visible simultaneously.

\subsection{Layout strategy}
The spatial arrangement of the tool and the spreadsheet has a significant impact on the cognitive load during the spreadsheet comprehension.
\begin{definition}
    \label{def:juxtaposability}
    Juxtaposability is the ability to see any two portions on screen side-by-side at the same time~\cite{green_usability_1996}.
\end{definition}
\subsubsection{The split view}
To achieve juxtaposability (Definition~\ref{def:juxtaposability}) between the spreadsheet and the flow graph, the system employs a split view layout. The spreadsheet occupies the left side, while the flow graph appears on the right side.
The split view layout introduces the cognitive cost known as the split-attention effect. Chandler and Sweller, in \textit{The Split-Attention Effect as a Factor in the Design of Instruction}~\cite{chandler_split-attention_1992}, describe this as a phenomenon in which learners are forced to mentally integrate information from disparate sources.
However, the split view is a response to a fundamental limitation of overlay-based approaches. As mentioned in section~\ref{sec:built-in-tools}, traditional spreadsheet tools for tracing precedents and dependents use an overlay strategy. An overlay strategy ties the dependency visualization directly to the current worksheet view. But it fails when the user navigates to another worksheet to inspect a referenced cell, the visualization disappears, causing the user to lose the visual context of the formula they were investigating. This is the loss of spatial context identified in section~\ref{sec:problem-statement}, where users lose track when navigating between dependents and precedents. Furthermore, an overlay would occlude cells and formulas the user needs to see. By decoupling the flow graph from worksheet navigation, the split view keeps the dependency graph of the synchronized formula visible and stable while the user is able to freely navigate between worksheets.

\subsection{Flow graph justification}
The decision to represent the dependency graph of a formula as a visual flow graph was motivated by the need to expose a structure that exists implicitly in every formula but remains hidden from the user. Rather than requiring the user to read and mentally parse individual formulas, a visual flow graph makes the dependencies directly visible. Hermans et al. demonstrate that dataflow diagrams help users understand spreadsheet structure across multiple levels of detail~\cite{hermans_supporting_2011}. The system in this thesis focuses on the formula level exclusively, as supporting multiple levels of abstraction would have added significant complexity. The flow direction is left to right, mirroring the spatial convention of spreadsheets where formula inputs are located in columns on the left side of a formula cell.

\subsection{Interaction design}
\label{sec:interaction-design}
Before interacting with the flow graph, the user must first synchronize it to a formula. When the user selects a cell containing a formula in the worksheet, the flow graph view exposes a synchronize button. Clicking it loads the selected formula's dependency graph into the flow graph view.
The flow graph serves as a visualization of a formula's dependency graph and as an interface for exploring and manipulating it. The interface operates in two modes, the preview mode and the edit mode, where edit mode extends the available interactions to include modifications to the underlying data. In edit mode the flow graphs border is orange whereas in preview mode it is a blue border. The visual distinction between modes serves as a cue to the user that changes are now possible. Edit Mode is intentionally easy to enter, prioritizing a low viscosity (definition~\ref{def:viscosity}). 

\subsubsection{Preview Mode}
Preview Mode is the default state of the flow graph and is designed for exploration. Hovering over a reference node highlights the corresponding cell or range in its worksheet. If the cell is already visible in the current viewport, this allows the user to see the dependencies at a glance without disrupting their current position in the worksheet. Clicking on a reference node explicitly scrolls to the corresponding worksheet and cell or range, enabling closer inspection when the user requests it. This aims to preserve the user's spatial context in the worksheet and only disrupt it on demand. 

\subsubsection{Edit Mode}
Edit Mode extends the interactions available in Preview Mode to allow direct manipulation of the flow graph and its underlying data. It is designed to minimize the distance between recognizing a value to change and modifying it. Rather than requiring the user to trace the node back to its source in a worksheet and modify it there, changes can be initiated directly from the point of interest in the flow graph, aiming to reduce viscosity~\cite{green_usability_1996}.
Edit Mode is always entered through an explicit user action:
\begin{samepage}
    \begin{enumerate}
    \item clicking on the edit button
    \item double-clicking on an inline value or address
    \item double-clicking an edge
    \item dragging from a connection handle
\end{enumerate}
\end{samepage}
The entry action determines the initial editing context. Double-clicking on a node's address allows the user to select a new address directly from the worksheet. This interaction is designed to leverage closeness of mapping to already known spreadsheet tools~\cite{green_usability_1996}. In  most spreadsheet tools the user can select cells and ranges directly in the workbook to supply them as function arguments. Double-clicking an inline value transforms the field into an editable input, enabling the user to modify the constant in place. In both cases, the change is committed by pressing Enter or clicking the confirmation button.
Double-clicking an edge enters Edit Mode with that edge pre-selected for further manipulation. Dragging from a connection handle initiates edge creation or modification. As the user drags from a source handle toward a target handle, invalid handles are grayed out to constrain the action to permissible connections. If the target handle has no existing connection, the new edge is created directly. If the target handle is already occupied, a popover prompts the user to either replace the existing edge or swap positions with it. The swap option aims to minimize the edit steps for especially switching arguments inside a function. Clicking the edit button enters the Edit Mode without an initial editing context. Once in edit mode, the user may freely select any element to begin editing.
