For the presentation layer there are 3 important goals:
\begin{enumerate}
    \item improve self-descriptiveness of the hidden data of the datatable
    \item improve bidirectionality between the tool and the datatable
    \item minimize the cognitive load of the user
\end{enumerate}

\subsection{Flow graph justification}
The decision to represent the hidden data as a flow graph was driven by the nature of the underlying state of spreadsheet formulas, the dependency graph.
A flow graph allows us to uncover this state and enables the user to see the dependencies rather than by reading formulas and parsing it on their own.
The flow direction is left to right. This should mirror the spatial convention for spreadsheets, where the inputs of a formula are located in columns on the left side of the formula.

\subsection{Interaction design}
The flow graph not only serves as a visualization but also as an interface for the exploration and manipulation of the datatable. Interactions are organized into two distinct modes: the \textbf{preview} mode and the \textbf{edit} mode. This separation should provide clear visual feedback when changes will affect the underlying data of the datatable.
\subsubsection{Preview Mode}
The Preview Mode is the default state and is intended for the exploration. Hovering over a reference node highlights the corresponding cell or range in the datatable. This allows the user to survey the dependency at the current position in the datatable. Clicking on a reference node scrolls the datatable to the corresponding cell or range to enable closer inspection of the dependencies.

Edit Mode is always entered through an explicit user action:
\begin{enumerate}
    \item clicking on the edit button
    \item double-clicking on an inline value or address
    \item double-clicking an edge
    \item dragging from a connection handle
\end{enumerate}
The entry action determines the initial editing context. Double-clicking on a node's address allows the user to select a new address directly from the datatable. The change is committed by pressing enter or clicking the confirmation symbol. Double-clicking an inline value transforms the field into an editable input. The user modifies the constant and commits the change through the same confirmation mechanism.
Double-clicking an edge enters the edit mode with the double-clicked edge selected for further manipulations. Dragging from a connection handle initiates edge creation or modification. The user drags from a source handle to a target handle. While dragging the edge invalid handles are grayed out. If the target handle has no existing connection, the new edge connects directly. Else the target handle already has a connection, a popover asks the user to either replace the existing edge or swap positions with it. Clicking only the edit button enters the edit mode without an initial editing context. Once in edit mode, the user can change context freely.
