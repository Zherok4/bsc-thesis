\section{The visualization layer}
\label{sec:visualization-layer}
In the visualization layer the system transforms the AST into a graph representation which is then rendered as an interactive flow-graph. This section focuses on the visual design choices of the flow-graph nodes and how each AST node maps to a corresponding graphical representation. 

The primary goal when transforming the AST is to reduce the overall number of nodes and edges while improving clarity. This aims to create a compact and informative flow-graph. Literal nodes are inlined into their parent nodes. Arithmetic operations are either transformed into a binary operation node or collapsed together into one expression node. Reference nodes are enhanced with visual cues and header inference heuristics. Function nodes are augmented with semantic argument labels. Conditional functions are represented by a separate node to help the comprehension of the branching logic. Function nodes as well as reference nodes display their intermediate results to provide more information about the current state to the user.

\subsection{Literals}
Literals are numbers or strings. They are used almost in every formula. Because the meaning of literals is evident, rendering them as nodes would not add meaningful help for the users. The system solves this by inlining literals into their parent node. There they are still visible but do not occupy any additional space in the graph.

\subsection{Arithmetic operations}
The system identifies certain operations as simple operations. These include arithmetic operators, string and comparison operations.
\subsubsection{Binary operation node}
If the left and right operand of these operations are not simple operations themselves, the system creates a binary operation node. Because these operations are self-explaining, the system aims to minimize the size of the node. Therefore, the system only displays the operator symbol with pseudo operators or inlined constants. 
\subsubsection{Expression nodes}
When multiple operations are chained together, the system creates an expandable expression node. Initially, this node displays only a preview of the complete expression, occupying minimal space in the flow-graph while still indicating to the user that here an operation is happening. Users can expand the node via double click to see the complete expression. 
Like the expandable cell reference node (section~\ref{sec: expandable cell references}) this design follows a details-on-demand strategy to balance compactness and accessibility. However, this introduces a trade-off: users who want to examine the complete expression must perform an additional step compared to a fully-expanded default state.
By grouping chained operations together, the system addresses two cognitive dimensions simultaneously. First, it reduces diffuseness~(definition~\ref{def:diffuseness}) by representing a chain of operations as a single node rather than multiple individual nodes. Second, it improves role~expressiveness~(definition~\ref{def:role-expressiveness}) by presenting these operations in a more familiar way. Because these simple operations are more self-explaining than functions, it is potentially easier for the user to read the familiar inline expression than a series of binary operation nodes.

\subsubsection{Label}
Hodnigg et al. identified in \textit{Comprehending Spreadsheets: Which Strategies do Users Apply?}~\cite{hodnigg_comprehending_2021}, the comprehension indicator "realization" which refers to the moment a user gains an insight about the spreadsheet. One of the comprehension patterns they identified is reading a label\footnote{textual descriptions in cells} followed by a "realization", indicating that labels can help in the process of comprehending spreadsheets. To support this, the system implements a header inference heuristic. One limitation of this heuristic is that it assumes well-structured spreadsheets and may yield suboptimal results in spreadsheets that do not follow standard column/row orientation. To generate a label, the system scans upwards and leftwards and selects the first non-formula string values to compose a combined label. For example, in figure \ref{fig:exp-cell-ref-m} the system resolved J4 to Bob:Rounded. This heuristic was chosen because most spreadsheets follow a structure with column headers above the data and row labels to the left. It is intentionally simple in order to cover the most common case. However, future work could extend it to handle less conventional layouts. By inferring the label for the user, the system aims to preserve the user's spatial context within the data table, as the users are no longer required to visually scan through columns and rows to understand what a cell represents. This was a pattern that contributed to the information detours identified by Ragavan et al.~\cite{srinivasa_ragavan_spreadsheet_2021}.

\subsubsection{Expandable Cell References}
\label{sec: expandable cell references}
As mentioned in section \ref{sec:problem-statement}, one major challenge for spreadsheet comprehension is the tracing of the dependency chain. To address this the system implements a recursive resolution strategy, to build the dependency chain.

%    An expandable cell reference is a precedent that itself contains a formula

To not overwhelm a user when presented with the whole dependency chain of the synchronized cell, the system adopts a details-on-demand, as described by Shneiderman \cite{shneiderman_eyes_2003} strategy. Upon synchronizing a new cell, the expandable cell reference nodes start in a minimized state (figure~\ref{fig:exp-cell-ref-m}). % TODO: reference a figure
Expandable cell reference nodes (figure~\ref{fig:exp-cell-ref-e}) are only expanded upon explicit user interaction. With this strategy the system aims to reduce the diffuseness of the flow-graph. 
Crucially, the graph navigation is decoupled from the grid navigation. When a user expands a cell reference node the viewport of the worksheet remains the same. This aims to preserve the spatial context of the user in the worksheet while being able to trace down the dependency chain, directly addressing an issue reported by Ragavan et al., where a user has lost track when navigating between dependents and precedents \cite{srinivasa_ragavan_spreadsheet_2021}. %TODO: mention even more why it helps keeping track of th positon => because the tool shows whole dependency chain and does stay the same => users can easily click on the graph to navigate through the worksheets without having to invest to much attention. ==> maybe write this in the edit section

\subsubsection{Merging references}
In spreadsheets, it is common that formulas use the same references. The flow-graph tries to merge same reference nodes, which are located in a distance of 5 edges, into one single node. The distance of 5 edges was chosen such that only nearby nodes can be merged and that identical nodes which are located at a completely different location do not disrupt the flow-graph by merging. Merging nodes together makes the flow-graph more compact.

\subsubsection{Range references}
In contrast to cell references, which hold one value, range references encompass many cells with many values. The system cannot display all the cells and their values of the range initially without cluttering the flow-graph. To reduce the diffuseness, the system implements the details-on-demand strategy, similar to the expression nodes and expandable reference nodes. If the user wants to know or edit a value of the range, they first must expand the range node. When expanded the user can edit values the same way as in cell reference nodes. 

\subsection{Functions}
\label{sec: functions}
A function in spreadsheet programs is a predefined formula, which takes one or more arguments to produce a result \cite{noauthor_ecma-376-12016_2016}. In traditional formula syntax, the semantic meaning of an argument is only defined by its position for example, in the function: VLOOKUP(E6,H6:I9,2,0). When reading this the user must actively recall which argument is responsible for what. This mental mapping increases cognitive load. To solve this the system gives each argument a semantic label. By visually linking a node to a semantic label the system aims to increase the role~expressiveness of the argument nodes. Additionally, this should decrease the need to consult documentation.

\subsubsection{Conditional Nodes}
Conditional functions\footnote{e.g IF, IFS} introduce branching logic to spreadsheets and add additional challenges for formula tracing.
First, as mentioned in the section above, conditional arguments also suffer from the positional semantics problem. Furthermore, conditional functions in textual form suffer from the lack of runtime visibility. Users must mentally simulate the boolean condition to determine which branch is active.
To address this, the system only displays the currently active branch and minimizes the inactive branches at first. This also helps to reduce the diffuseness of the flow-graph but hides some logic as tradeoff.
To highlight the active branch to the user, the system implements the following strategies:
\begin{samepage}
    \begin{enumerate}
    \item Minimization of inactive branches
    \item Visual dimming
    \item Highlighting the contributing argument
\end{enumerate}
\end{samepage}
As mentioned above, the system minimizes inactive branches by default. If the users still want to explore the logic of the inactive branch they can expand the subflow by clicking on the plus symbol. This mechanic is identical to the one of expandable cell references as mentioned in section \ref{sec: expandable cell references}. In order to still have a visual difference between the active and inactive branches, when the user expands inactive branches, the system dims the inactive branches (figure~\ref{fig:conditional-node-branch-gray}). To reinforce this state in the node the inactive arguments are struck through (figure~\ref{fig:conditional-node-arguments}). These visual cues aim to prevent misinterpreting an inactive value as a contributing factor to the result of the conditional node.
Furthermore, because spreadsheets are dynamic, the system updates the active branch when dependents change.

\subsubsection{Help Button}
Ragavan et al. identify the "understanding of unfamiliar concepts"~\cite[p.~11]{srinivasa_ragavan_spreadsheet_2021} as one of the key information needs. This need arises when users encounter unfamiliar functions and do not understand what they calculate. Every function node has a help button, which provides a summary of what the function calculates based on its input values and result, to provide the user an intermediate step before having to consult the documentation. By offering this summary directly in the flow graph, the system aims to reduce the need to consult external documentation, which helps reduce information detours. For future work, the help button could be extended to provide explanations when errors occur in a formula, to help diagnose the cause.


\subsection{Intermediate Results}
Hodnigg et al. also identified a comprehension pattern in which reading values leads to realizations \cite{hodnigg_comprehending_2021}. This motivated the decision to display the intermediate results of reference nodes and functions. Intermediate results in reference nodes refer to the content of the cell or range, while in function nodes they refer to the calculated output of the function. The second reason the system explicitly shows the intermediate results is to allow the users to verify each sub-step in the flow-graph.
